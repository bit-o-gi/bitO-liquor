# Lean PRD

## 문제
현재 카탈로그 화면은 동일한 주류가 여러 행으로 내려오는 것을 전제로 프론트엔드에서 `name` 기준으로 다시 묶고, 벤더별 가격 목록을 조합해 화면에 노출한다.

하지만 DB에서 중복 상품을 제거하고, 하나의 주류 엔티티에 대해 벤더별 가격만 별도로 관리하는 구조로 정리되었다. 이 변경이 프론트와 API 응답 구조에 반영되지 않으면, 화면에서 중복 제거나 판매처 가격 노출이 실제 데이터 구조와 어긋날 수 있다.

## 목표
- DB의 중복 제거 및 벤더별 가격 분리 구조를 현재 카탈로그 API와 프론트 화면에 반영한다.
- 화면에서 하나의 주류 카드에 여러 판매처 가격이 정확히 매핑되도록 정리한다.
- 이번 변경의 범위와 진행 상태를 `008.catalog-deduped-vendor-pricing-sync` 문서에서 관리한다.

## 비목표
- 카탈로그 레이아웃 자체를 다시 전면 개편하지 않는다.
- 검색 UX, 무한 스크롤 UX를 새로 설계하지 않는다.
- 크롤러 수집 로직 자체를 이번 이슈에서 직접 수정하는 것을 기본 목표로 삼지 않는다.

## 사용자 가치
- 같은 상품이 중복 카드로 보이지 않고, 하나의 카드 안에서 판매처별 가격 비교가 일관되게 제공된다.
- 실제 DB 구조와 UI 표시가 일치해 가격 정보 신뢰도가 올라간다.
- 이후 가격 집계, 정렬, 검색 개선 작업을 더 안정적으로 이어갈 수 있다.

## 핵심 요구사항
- DB의 최신 구조 기준으로 주류 기본 정보와 벤더별 가격 데이터를 어떤 단위로 내려주는지 확인해야 한다.
- API 응답이 기존 평면 리스트인지, 주류 + vendorPrices 형태의 집계 응답인지 명확히 정리해야 한다.
- 프론트는 더 이상 잘못된 기준으로 중복 상품을 다시 묶지 않도록 조정해야 한다.
- 주류 카드의 최저가, 판매처 수, 판매처별 링크/가격이 실제 데이터 구조와 맞아야 한다.
- 작업 중 결정한 데이터 계약과 화면 반영 방식은 문서에 즉시 기록해야 한다.

## 성공 기준
- 카탈로그에서 중복 상품 노출 없이 하나의 상품 카드에 벤더 가격이 정확히 연결된다.
- API 응답 구조와 프론트 모델이 현재 DB 구조와 일치한다.
- 변경 후 기본 검증 명령과 핵심 화면 확인 결과가 문서에 남는다.

## 범위
포함:
- 관련 DB 조회 구조 확인
- 카탈로그 API 응답 구조 정리
- 프론트 도메인 모델 및 그룹핑 로직 조정
- 문서 갱신

제외:
- 신규 디자인 작업
- 비카탈로그 페이지 변경
- 대규모 DB 마이그레이션 설계 문서화 전반

## 제약 및 전제
- 현재 Next.js 프론트와 Supabase 조회 경로를 우선 기준으로 분석한다.
- 백엔드 Spring API가 아니라 `frontend`의 서버 데이터 패치 경로가 실제 카탈로그 진입점일 수 있다.
- DB 구조 변경이 이미 적용되었다는 전제하에, 우선은 조회 계약과 화면 반영 불일치를 해소한다.

## 오픈 질문
- 현재 Supabase에서 사용 중인 테이블 또는 뷰가 중복 제거 후 구조를 이미 반영했는지 확인이 필요하다.
- 벤더 가격을 API에서 집계해 내려줄지, 프론트에서 조합할지를 최종 결정해야 한다.
