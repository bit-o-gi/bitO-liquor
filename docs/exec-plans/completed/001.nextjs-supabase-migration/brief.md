# Lean PRD

## 문제
현재 서비스는 프론트엔드가 Spring `backend/api`를 통해 조회·검색·추천 데이터를 받아오고, 별도의 `backend/crawler`가 크롤링을 수행하는 구조다. 앞으로는 백엔드 역할을 단순화해 크롤링에 집중시키고, 애플리케이션의 주 데이터 접근 경로는 Next.js와 Supabase 중심으로 재구성하려고 한다.

이 구조를 바꾸지 않으면 조회 API, 추천 API, 크롤링 파이프라인이 하나의 백엔드 문맥에 섞여 유지보수 비용이 계속 커진다. 프론트 전환과 데이터 계층 전환도 동시에 설계하지 않으면 책임 경계가 다시 불분명해질 가능성이 높다.

## 목표
- 프론트엔드의 주 데이터 접근 경로를 Spring API가 아니라 Next.js + Supabase 중심으로 전환한다.
- 기존 Spring 백엔드는 크롤링과 데이터 적재 역할에 집중하도록 범위를 축소한다.
- 전환 과정에서 현재 주류 목록 조회와 검색 경험이 끊기지 않도록 단계적 이전 경로를 마련한다.

## 비목표
- 이번 이슈에서 전체 UI를 새로 디자인하지 않는다.
- 크롤링 대상 사이트나 크롤링 방식 자체를 전면 개편하지 않는다.
- Supabase 도입과 동시에 모든 운영 자동화, 배포 파이프라인까지 완성하는 것을 목표로 하지 않는다.

## 사용자 가치
- 데이터 조회 구조가 단순해져 이후 기능 추가와 유지보수가 쉬워진다.
- 프론트엔드가 데이터 계층과 더 가깝게 연결되어 화면 개발 속도가 빨라진다.
- 크롤링 백엔드의 책임이 명확해져 장애 원인과 수정 범위를 빠르게 파악할 수 있다.

## 핵심 요구사항
- 주류 목록 조회와 검색은 Supabase 기반으로 이전 가능해야 한다.
- 프론트엔드는 이번 이슈에서 Vite React가 아니라 Next.js로 실제 전환되어야 한다.
- 기존 추천 기능은 이전하지 않고 제거 범위와 제거 시점을 명확히 해야 한다.
- 크롤러는 계속 동작해야 하며, 최종적으로는 조회 API가 아니라 데이터 수집 및 적재에 집중해야 한다.
- 문서 기준으로 목표 아키텍처, 구현 계획, 진행 상황이 지속적으로 관리되어야 한다.

## 성공 기준
- 프론트엔드가 더 이상 Spring `backend/api` 조회 엔드포인트에 직접 의존하지 않는다.
- 프론트엔드 실행 기반이 Vite React에서 Next.js로 전환된다.
- 크롤링 관련 기능은 `backend/crawler` 중심으로 남고, 조회 중심 책임은 분리된다.
- 목록 조회, 검색, 추천 제거 범위와 단계가 문서로 명확히 정리된다.
- 전환 작업 중 현재 서비스 핵심 흐름이 중단되지 않는다.

## 범위
포함:
- Next.js + Supabase 중심 구조로의 전환 설계
- 기존 Spring API 책임 분석 및 축소 계획 수립
- 주류 조회/검색 기능의 이전 경로 정의
- 기존 추천 기능 제거 계획 수립 및 반영
- 크롤러와 데이터 저장소 역할 재정의

제외:
- 크롤링 대상 확대
- 브랜딩 또는 대규모 UI 리뉴얼
- 운영 자동화 전반의 완성

## 제약 및 전제
- 현재 저장소 프론트엔드는 Vite React 기반이므로, Next.js 전환은 데이터 계층 전환과 별개의 실제 마이그레이션 작업이 필요하다.
- 현재 추천 로직은 Spring 서비스 내부 계산 로직에 존재하지만, 이번 이슈에서는 동등 이전 대신 기능 제거를 선택한다.
- 기본 이미지는 현재 사용 중인 Supabase Storage 공개 경로 fallback을 유지하고, 이번 이슈에서는 별도 이미지 정책 변경을 하지 않는다.
- `backend/crawler`의 적재는 별도 중간 API 없이 Supabase가 연결된 Postgres에 JPA로 직접 쓰는 방향을 기준으로 한다.
