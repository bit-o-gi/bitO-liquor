# Lean PRD

## 문제
현재 프론트엔드는 Next.js App Router 전환과 UI 회귀 복구를 거치며 기능적으로는 안정화됐지만, 구조는 여전히 `components`, `api`, `lib`, `utils`, `types` 수준의 얕은 분리만 적용된 상태다.

이 구조는 현재 규모에서는 동작하지만, 카탈로그 기능에 조회 규칙, 검색 상태, 페이징, 데이터 매핑, 외부 데이터 소스 의존성이 함께 쌓이기 시작하면 변경 영향 범위를 빠르게 넓힌다. 특히 UI 컴포넌트가 조회 규칙과 에러 처리, 페이징 흐름을 직접 소유하는 방식은 이후 기능 추가 시 테스트성과 유지보수성을 떨어뜨릴 가능성이 높다.

지금 시점에서 프론트 구조를 정리하지 않으면, 이후 기능 확장 때마다 화면 단위로 로직이 비대해지고 데이터 소스 변경 비용도 커질 수 있다.

## 목표
- 프론트엔드에 점진적으로 적용 가능한 클린 아키텍처 기준 구조를 정의한다.
- 카탈로그 기능을 시작점으로 UI, 유스케이스, 데이터 접근 책임을 분리한다.
- 현재 동작 중인 검색, 목록 조회, 무한 스크롤, 초기 서버 선조회 흐름은 유지한다.
- 이후 신규 기능이 같은 구조를 따를 수 있도록 최소 규칙과 경계를 정리한다.

## 비목표
- 프론트 전체를 한 번에 전면 재작성하지 않는다.
- 디자인 개편이나 사용자 경험 변경을 이번 이슈의 목표로 삼지 않는다.
- Supabase 스키마, 내부 API 스펙, 백엔드 계약을 변경하지 않는다.
- 추천 기능이나 신규 도메인 기능을 이번 이슈 범위에 포함하지 않는다.

## 사용자 가치
- 기능 추가와 구조 개선이 동시에 진행되더라도 화면 품질과 동작 안정성이 덜 흔들린다.
- 데이터 소스나 조회 규칙이 바뀌어도 UI 수정 범위를 줄일 수 있다.
- 팀이 같은 기준으로 기능을 추가할 수 있어 구조 일관성이 높아진다.

## 핵심 요구사항
- 카탈로그 기능부터 얇은 클린 아키텍처 구조를 적용해야 한다.
- 도메인 모델과 외부 응답 DTO를 분리해야 한다.
- 데이터 접근 책임은 repository 계층 또는 이에 준하는 경계로 모아야 한다.
- 검색/페이지네이션/목록 조합 규칙은 usecase 또는 feature model 계층으로 이동해야 한다.
- React 컴포넌트는 화면 렌더링과 사용자 입력 처리에 집중하고, 조회 orchestration 로직은 별도 훅 또는 모델 계층으로 분리해야 한다.
- 기존 Next.js App Router, 내부 API route, Supabase 기반 조회 흐름은 유지해야 한다.

## 성공 기준
- 카탈로그 기능의 주요 파일들이 UI, 유스케이스, 데이터 접근 책임 기준으로 재배치된다.
- `CatalogPageClient` 수준의 화면 컴포넌트가 직접 조회 상세 로직을 소유하지 않게 된다.
- 현재 검색, 목록 조회, 무한 스크롤, 초기 렌더링 동작이 회귀 없이 유지된다.
- 이후 기능 추가 시 참고할 수 있는 폴더 구조와 의존성 규칙이 문서화된다.
- 최소 `npm run lint`, `npm run build`, `npm run test:e2e` 검증을 통과한다.

## 범위
포함:
- 카탈로그 기능의 구조 재배치
- UI 계층과 조회 orchestration 계층 분리
- 데이터 매핑과 저장소 접근 경계 정리
- 전환 기준 문서화 및 검증

제외:
- 프론트 전체 기능 일괄 전환
- 백엔드 API/DB 스키마 변경
- 디자인 리뉴얼
- 신규 기능 추가

## 제약 및 전제
- 현재 프론트 규모를 고려할 때 교과서식 풀 레이어 도입은 과할 수 있으므로, 얇고 실용적인 구조를 우선한다.
- 신규 구조는 기존 코드와 일정 기간 공존할 수 있어야 하며, 단계적 이전을 허용해야 한다.
- 기존 검증 자산과 동작 계약을 유지하는 방향으로 리팩터링해야 한다.
- 파일 수를 늘리는 것 자체가 목적이 아니며, 변경 비용을 줄이는 경계 설정이 목적이다.

## 오픈 질문
- feature 단위 구조를 `features/catalog` 중심으로 둘지, `entities / features / shared` 조합으로 시작할지 최종 기준을 어떻게 확정할지 검토가 필요하다.
- 내부 API route 호출과 서버 선조회 로직을 동일 usecase로 추상화할지, 서버/클라이언트 진입점을 분리할지 결정이 필요하다.
