# Product Requirements Document (PRD)

## 1. 문서 목적

이 문서는 Jururuk의 제품 요구사항(Why/What)을 정의한다.

- 포함: 문제 정의, 목표, 사용자, 핵심 시나리오, 기능 요구사항, 수용 기준
- 제외: 코드 구조, API 구현 디테일, 엔티티 스키마, 알고리즘 상세

## 2. 제품 비전

Jururuk은 위스키 탐색과 구매 결정을 빠르게 돕는 취향 기반 추천 서비스다.

핵심 가치:
- 취향을 모르는 사용자도 빠르게 후보를 찾는다.
- 추천과 실제 구매 정보(가격/판매처)를 한 흐름으로 연결한다.
- 반복 탐색 없이 Top 후보를 제시해 의사결정 시간을 줄인다.

## 3. 문제 정의

현재 사용자 문제:
- 위스키 선택 기준이 없어 탐색 피로가 크다.
- 상품명/브랜드가 많아 비교가 어렵다.
- 추천 콘텐츠와 구매 경로가 분리되어 있다.

## 4. 목표 / 비목표

### 4.1 목표 (Goals)

- G1. 검색 가능한 위스키 카탈로그를 제공한다.
- G2. 취향 테스트 기반 Top 5 추천을 제공한다.
- G3. 추천 결과에서 구매 전환(판매처 이동)을 지원한다.
- G4. 데이터 최신성을 유지할 운영 플로우를 갖춘다.

### 4.2 비목표 (Non-Goals)

- N1. 회원 시스템(로그인/권한/마이페이지)
- N2. 결제/주문/장바구니
- N3. 사용자 설문 응답 이력의 장기 보관

## 5. 대상 사용자

- 위스키 입문자: 취향 기반 첫 추천이 필요한 사용자
- 탐색형 사용자: 브랜드/스타일/가격을 빠르게 비교하려는 사용자
- 가성비 지향 사용자: 추천과 동시에 구매처를 확인하려는 사용자

## 6. 핵심 사용자 시나리오

### 시나리오 A: 카탈로그 탐색

1) 사용자는 메인 페이지에서 위스키 목록을 본다.
2) 사용자는 키워드로 검색해 범위를 좁힌다.
3) 사용자는 후보 제품의 가격/판매처 정보를 확인한다.

### 시나리오 B: 취향 테스트 기반 추천

1) 사용자는 취향 테스트를 시작한다.
2) 시스템은 답변을 flavor profile로 변환한다.
3) 시스템은 Top 5 추천과 추천 이유를 제공한다.
4) 사용자는 결과를 보고 재테스트하거나 홈으로 돌아간다.

### 시나리오 C: 구매 전환

1) 사용자는 추천 카드의 판매처 버튼을 누른다.
2) 외부 판매 페이지로 이동해 구매를 이어간다.

## 7. 기능 요구사항 (Functional Requirements)

### FR-01. 카탈로그 조회

- 사용자는 메인에서 위스키 목록을 볼 수 있어야 한다.
- 목록은 사용자가 비교 가능한 형태(이름/브랜드/카테고리/가격/판매처)로 제공되어야 한다.

### FR-02. 검색

- 사용자는 제품명 또는 브랜드로 목록을 필터링할 수 있어야 한다.
- 검색 결과가 없을 경우 빈 상태 메시지를 제공해야 한다.

### FR-03. 취향 테스트

- 테스트는 10문항 단일 선택으로 구성되어야 한다.
- 진행률이 명확히 표시되어야 한다.
- 마지막 문항 완료 시 추천 요청이 실행되어야 한다.

### FR-04. 추천 결과

- 시스템은 사용자 취향에 맞는 위스키 Top 5를 제공해야 한다.
- 각 추천 항목은 최소 다음 정보를 포함해야 한다:
  - 제품 기본 정보
  - 가격 및 판매처
  - 추천 이유
  - 유사도 지표

### FR-05. 추천 결과 UX

- 사용자는 결과 화면에서 다음 액션을 수행할 수 있어야 한다:
  - 테스트 다시 하기
  - 홈으로 돌아가기
- 홈 복귀 시 메인 맥락(탐색 진입 상태)이 자연스럽게 회복되어야 한다.

### FR-06. 데이터 운영

- 운영자는 데이터 갱신 플로우를 실행할 수 있어야 한다.
- 데이터 갱신 실패 시 원인 추적이 가능한 수준의 피드백이 필요하다.

## 8. 비기능 요구사항 (NFR)

- NFR-01. 안정성: 외부 API/데이터 실패 상황에서도 서비스가 완전히 중단되지 않아야 한다.
- NFR-02. 사용성: 핵심 플로우(탐색/테스트/결과)가 모바일/데스크톱 모두에서 사용 가능해야 한다.
- NFR-03. 응답성: 일반 사용 환경에서 목록/추천 체감 응답은 지연이 과도하지 않아야 한다.
- NFR-04. 유지보수성: 기능 추가 시 요구사항 변경점을 문서에서 추적 가능해야 한다.

## 9. 성공 지표 (Success Metrics)

- M1. 추천 요청 성공률
- M2. 테스트 완료율
- M3. 추천 결과 클릭률(판매처 이동)
- M4. 검색 사용률 및 검색 후 클릭률

## 10. 리스크 및 대응

- R1. 데이터 소스 품질 편차로 추천 신뢰도 저하
  - 대응: 품질 검증 규칙과 보강 전략 운영
- R2. 외부 채널 구조 변경으로 데이터 갱신 실패
  - 대응: 모니터링 및 빠른 수정 프로세스 확보
- R3. 사용자 기대와 추천 결과 간 갭
  - 대응: 추천 이유 설명 강화 및 재탐색 UX 개선

## 11. 오픈 이슈

- O1. 사용자 피드백(좋아요/싫어요) 기반 추천 고도화 범위
- O2. 가격 비교 기준(최저가/신뢰도/신선도) 우선순위 정책
- O3. 향후 개인화 저장 기능 도입 여부

## 12. 수용 기준 (Acceptance Criteria)

- AC-01. 사용자는 메인에서 위스키 목록을 조회하고 검색할 수 있다.
- AC-02. 사용자는 10문항 테스트를 완료하고 추천 Top 5를 확인할 수 있다.
- AC-03. 추천 결과에서 판매처 이동이 가능하다.
- AC-04. 추천 결과에는 추천 이유와 유사도 정보가 표시된다.
- AC-05. 홈 복귀 동작은 사용자가 다시 탐색을 시작하기 쉬운 상태를 제공한다.

## 13. PRD 업데이트 원칙

- 업데이트 필요:
  - 제품 목표/범위 변경
  - 핵심 사용자 플로우 변경
  - 수용 기준 변경
- 업데이트 불필요:
  - 단순 버그 수정
  - 내부 리팩터링
  - 구현 상세 변경(요구사항 불변)
