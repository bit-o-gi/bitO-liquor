# Lean PRD

## 문제
현재 카탈로그 검색은 입력 후 결과가 느리게 반응하는 체감이 있으며, 특히 부분 일치 검색에서 응답 속도가 충분히 안정적이지 않다.

현재 검색은 상품명, 정규화명, 브랜드에 대한 contains 검색 성격이 강한데, 이 패턴은 기본 인덱스로는 효율이 낮아 데이터가 늘수록 성능 저하가 커질 가능성이 있다.

## 목표
- 카탈로그 검색 성능 개선 작업을 별도 이슈로 분리한다.
- PostgreSQL `pg_trgm` 기반 인덱싱을 중심으로 부분 일치 검색 성능 개선 방향을 정리한다.
- 현재 사용자 검색 경험을 유지하면서 검색 응답 시간을 줄일 수 있는 구현 범위를 정의한다.

## 비목표
- 검색 UI를 전면 개편하지 않는다.
- 추천, 자동완성, 오타 교정 같은 새 검색 기능을 추가하지 않는다.
- 카탈로그 전체 아키텍처를 다시 바꾸지 않는다.
- 이번 문서 단계에서 곧바로 DB 스키마 전체를 재설계하지 않는다.

## 사용자 가치
- 검색어 입력 후 결과가 더 빠르게 반응한다.
- 상품명 일부만 입력해도 현재 검색 경험을 유지하면서 더 안정적인 응답을 기대할 수 있다.
- 검색 성능 개선이 별도 이슈로 관리되어 변경 이유와 범위를 추적하기 쉬워진다.

## 핵심 요구사항
- 검색 성능 이슈는 `006.search-performance-pg-trgm` 문서 범위에서 관리한다.
- 검색 대상인 상품명, 정규화명, 브랜드의 부분 일치 검색 성능 개선 방안을 `pg_trgm` 기준으로 검토한다.
- 현재 검색 동작의 의미론은 유지해야 한다.
  현재처럼 부분 문자열 검색이 가능해야 한다.
- 필요 시 검색 쿼리와 인덱스 전략을 함께 조정할 수 있어야 한다.
- 필요 시 전체 개수 계산 방식과 가격 조회 방식이 검색 지연에 주는 영향도 함께 검토한다.

## 성공 기준
- `pg_trgm` 적용 후 현재 검색 패턴에서 성능 개선 기대 지점이 명확히 설명된다.
- 구현 단계에서는 부분 검색 기능 회귀 없이 응답 시간이 개선되어야 한다.
- 변경 범위와 이유가 `006` 문서 안에서 추적 가능해야 한다.

## 범위
포함:
- 검색 성능 병목 원인 정리
- `pg_trgm` extension 및 trigram 인덱스 적용 검토
- 검색 쿼리와 인덱스 호환성 검토
- 필요 시 count 처리 방식 검토

제외:
- 추천 검색 도입
- 검색 결과 정렬 정책 대개편
- 카탈로그 외 다른 도메인 검색 최적화
- UI/브랜딩 작업

## 제약 및 전제
- 현재 검색은 부분 일치 검색을 유지해야 한다.
- PostgreSQL 또는 Supabase 환경에서 적용 가능한 방식이어야 한다.
- 성능 개선은 체감뿐 아니라 실제 쿼리 비용 관점에서도 설명 가능해야 한다.

## 오픈 질문
- `pg_trgm`만으로 충분한지, 아니면 `count: exact` 제거와 최신 가격 조회 구조 개선을 함께 묶어야 하는지 구현 단계에서 판단이 필요하다.
