# Lean PRD

## 문제
현재 메인 카탈로그 화면은 기존 레이아웃 기준으로 구성되어 있어, 최근 Stitch에서 만든 새 디자인 시안의 구조와 분위기를 반영하지 못하고 있다.

디자인 업데이트를 별도 이슈로 분리하지 않으면 UI 레이아웃 변경과 기능 수정 이력이 섞여, 어떤 범위까지 화면 개편이 이뤄졌는지 추적하기 어려워진다.

## 목표
- Stitch 기반의 새로운 레이아웃을 현재 메인 카탈로그 화면에 적용한다.
- 기존 검색, 초기 로딩, 무한 스크롤 같은 핵심 카탈로그 동작은 유지한 채 디자인만 새 구조에 맞게 갱신한다.
- 이번 변경의 범위와 진행 상태를 `007.stitch-layout-design-update` 문서에서 관리한다.

## 비목표
- 백엔드 API, DB 스키마, 검색 로직을 새로 설계하지 않는다.
- 카탈로그 외 다른 페이지까지 한 번에 전면 개편하지 않는다.
- 추천, 필터, 정렬 같은 새 기능을 이번 이슈 범위에 추가하지 않는다.

## 사용자 가치
- 첫 화면에서 더 완성도 높은 브랜드 인상과 정보 구조를 제공할 수 있다.
- 상품 탐색 화면의 위계와 가독성이 좋아져 탐색 피로가 줄어든다.
- 디자인 업데이트 범위가 별도 이슈로 관리되어 회귀 추적과 후속 수정이 쉬워진다.

## 핵심 요구사항
- Stitch 시안의 레이아웃 방향을 현재 `frontend` 메인 카탈로그 진입 화면에 반영해야 한다.
- 검색 입력, 상품 목록 노출, 추가 페이지 로딩, 오류/재시도 같은 기존 핵심 상호작용은 유지해야 한다.
- 데스크톱과 모바일 모두에서 레이아웃이 무너지지 않아야 한다.
- 디자인 적용 과정에서 현재 코드베이스 구조에 맞지 않는 과도한 마크업이나 스타일은 정리할 수 있어야 한다.
- 작업 범위, 구현 계획, 진행 상태는 `007` 문서 세트에 계속 기록해야 한다.

## 성공 기준
- 메인 카탈로그 레이아웃이 Stitch 시안 기준의 새 구조와 시각 톤으로 갱신된다.
- 기존 검색 및 목록 탐색 흐름에 기능 회귀가 없다.
- `frontend` 기준 기본 검증 명령이 통과하고, 문서에 결과가 남는다.

## 범위
포함:
- 메인 카탈로그 레이아웃 및 시각 구조 업데이트
- 필요 시 관련 UI 컴포넌트 분리/정리
- 정적 자산, 스타일, 문서 갱신

제외:
- 백엔드/DB 변경
- 신규 검색 기능 추가
- 다른 페이지 동시 개편

## 제약 및 전제
- 현재 Next.js/Tailwind 기반 구조 안에서 구현한다.
- 서버 데이터 계약은 유지하고, UI 계층에서 레이아웃 변경을 우선 처리한다.
- Stitch 산출물이 그대로 들어가기보다 현재 코드 품질 기준에 맞게 정리되어야 한다.

## 오픈 질문
- Stitch 산출물이 시안 기준인지, 바로 가져온 코드 기준인지에 따라 정리 범위가 달라질 수 있다.
