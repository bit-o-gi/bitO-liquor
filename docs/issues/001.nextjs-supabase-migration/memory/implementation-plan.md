# Implementation Plan

## 상태 규칙
- `[ ]` 미착수
- `[-]` 진행 중
- `[x]` 완료
- 작업 진행 시 체크 상태를 즉시 갱신하고, 완료 기준 충족 시 해당 phase와 세부 작업을 함께 업데이트한다.

## 방향
- 목표 아키텍처는 `frontend`의 주 데이터 접근 경로를 Spring `backend/api`에서 분리하고, Next.js + Supabase를 조회 중심 계층으로 두는 것이다.
- `backend/crawler`는 데이터 수집과 적재 책임에 집중하고, 조회/검색/추천을 위한 API 책임은 점진적으로 축소한다.
- 사용자 영향이 큰 목록 조회와 검색을 먼저 이전하고, 추천은 로직 위치를 명확히 결정한 뒤 후속 phase에서 이전한다.
- 전환 중 서비스 중단을 피하기 위해 기능별 병행 운영 기간을 둔다.
- 구현은 TDD를 기본 원칙으로 두고, 각 phase에서 실패하는 테스트를 먼저 추가한 뒤 구현과 리팩터링을 진행한다.

## [ ] Phase 1. 현행 구조 분석 및 목표 아키텍처 확정
### 목표
- 현재 `backend/api`, `backend/crawler`, `frontend` 사이의 책임 경계를 문서화하고 최종 전환 범위를 고정한다.

### 작업
- [ ] 현재 목록 조회, 검색, 추천에 사용되는 Spring API 엔드포인트와 서비스/리포지토리 의존성을 식별한다.
- [ ] 프론트엔드에서 `backend/api`를 호출하는 화면, API 어댑터, 타입 경로를 정리한다.
- [ ] Supabase를 조회 계층으로 사용할 때 필요한 테이블, 뷰, 인덱스, Storage 연계 지점을 정리한다.
- [ ] 이후 phase에서 테스트를 먼저 작성할 수 있도록 기능별 기대 동작과 회귀 기준을 테스트 관점으로 정리한다.
- [ ] Next.js 전환 범위를 확정한다.
  선택지 A: 기존 Vite React를 단기 유지하고 Supabase 연동부터 이전
  선택지 B: Next.js 앱으로 화면과 데이터 계층을 함께 이전
- [ ] 추천 로직의 이전 후보를 비교한다.
  Next.js 서버 로직
  Supabase RPC / Edge Functions
  일부 `backend/api` 임시 유지

### 산출물
- 책임 분리 기준이 반영된 목표 아키텍처 문서
- 기능별 이전 우선순위 표
- 추천 로직과 프론트 전환 방식에 대한 의사결정 기록

### 완료 기준
- 어떤 기능이 어느 런타임으로 이동하는지 문서만 봐도 판단 가능해야 한다.
- 이후 phase가 의존할 데이터 경로와 책임 경계가 고정되어야 한다.

## [ ] Phase 2. Supabase 데이터 계층 준비
### 목표
- 목록 조회와 검색을 안정적으로 이전할 수 있도록 Supabase 스키마, 접근 정책, 적재 경로를 준비한다.

### 작업
- [ ] 기존 조회 API가 사용하는 주요 엔티티를 Supabase 테이블/뷰 기준으로 매핑한다.
- [ ] 목록 조회와 검색 성능을 고려한 인덱스, 정렬 기준, 페이징 정책을 정의한다.
- [ ] 이미지 및 자산 경로가 있다면 Supabase Storage 참조 정책을 정리한다.
- [ ] 데이터 계층 변경 전에 쿼리 결과, 정렬, 필터링 규칙을 검증하는 테스트 케이스를 먼저 정의한다.
- [ ] 크롤러 적재 경로를 결정한다.
  `backend/crawler`가 Supabase에 직접 적재할지
  중간 적재 계층을 둘지
- [ ] 읽기/쓰기 권한 모델을 정의한다.
  프론트는 조회 중심
  크롤러는 적재 중심

### 산출물
- Supabase 스키마/뷰 설계안
- 크롤러 적재 방식 결정
- 목록/검색용 쿼리 규격 초안

### 완료 기준
- 목록 조회와 검색이 Spring API 없이도 가능한 데이터 구조가 준비되어야 한다.
- 크롤러 적재 경로와 권한 정책이 문서화되어야 한다.

## [ ] Phase 3. 목록 조회/검색 기능 이전
### 목표
- 사용자 핵심 흐름인 목록 조회와 검색을 Supabase 기반으로 먼저 이전한다.

### 작업
- [ ] 목록 조회/검색의 현재 기대 동작을 보존하는 실패 테스트를 먼저 작성한다.
- [ ] 프론트엔드 데이터 접근 계층을 교체한다.
  Next.js 채택 시 서버 컴포넌트, 라우트 핸들러, 서버 액션 등 적절한 계층에서 조회 처리
  Vite 유지 시 Supabase 클라이언트 기반 조회 어댑터로 우선 전환
- [ ] 기존 API 응답과 화면 요구사항을 비교해 필요한 타입/DTO 변환 규칙을 정리한다.
- [ ] 목록, 검색, 상세 진입 흐름의 회귀 테스트 포인트를 정의한다.
- [ ] 병행 운영 기간 동안 기능 플래그 또는 환경별 API 스위칭 전략을 둔다.

### 산출물
- Supabase 기반 목록/검색 데이터 접근 구현
- 화면 회귀 체크리스트
- 단계적 전환 스위칭 전략

### 완료 기준
- 프론트엔드의 목록 조회와 검색이 더 이상 `backend/api` 조회 엔드포인트에 직접 의존하지 않아야 한다.
- 핵심 화면 흐름이 기존 수준으로 유지되어야 한다.

## [ ] Phase 4. 추천 기능 이전 및 책임 정리
### 목표
- 추천 기능 품질을 유지하면서 실행 위치를 새 구조에 맞게 재배치한다.

### 작업
- [ ] 현재 Spring 서비스 내부 추천 계산 로직을 분석해 입력값, 계산 규칙, 데이터 의존성을 정리한다.
- [ ] 추천 품질과 응답 규칙을 고정하는 테스트를 먼저 작성해 이전 전후 결과를 비교 가능하게 만든다.
- [ ] 선택된 이전 위치에 따라 실행 방식을 구현한다.
  Next.js 서버 로직이면 서버 전용 서비스 모듈화
  Supabase 함수/RPC면 DB 연산 또는 함수 경계 설계
  임시 유지면 제거 시점과 대체 조건 명시
- [ ] 추천 결과 검증 기준을 정의한다.
  기존 응답과의 비교
  품질 저하 여부 확인
- [ ] 추천 API 소비 화면을 새 경로로 전환한다.

### 산출물
- 추천 로직 이전 구현안
- 품질 비교 기준 및 검증 결과
- 임시 잔존 책임이 있으면 제거 계획

### 완료 기준
- 추천 기능의 실행 위치와 유지 책임이 명확해야 한다.
- 추천 경험이 전환 전 대비 실질적으로 깨지지 않아야 한다.

## [ ] Phase 5. Spring API 책임 축소 및 운영 정리
### 목표
- `backend/api`의 조회 중심 책임을 제거하거나 최소화하고, `backend/crawler` 중심 구조로 정리한다.

### 작업
- [ ] Spring API 축소 전에 남겨야 하는 계약과 제거 가능한 계약을 테스트로 분리해 안전망을 만든다.
- [ ] 더 이상 사용하지 않는 조회/검색 API를 비활성화하거나 제거한다.
- [ ] `backend/api`에 남아야 하는 기능과 제거 가능한 기능을 구분한다.
- [ ] 크롤러 운영에 필요한 설정, 적재 실패 대응, 모니터링 포인트를 정리한다.
- [ ] 전환 이후 문서, 환경변수, 로컬 실행 방법을 최신 구조 기준으로 갱신한다.

### 산출물
- 축소된 Spring API 책임 목록
- 운영 전환 체크리스트
- 최신 개발/운영 문서

### 완료 기준
- 조회 중심 책임이 새 계층으로 이동했고, Spring 백엔드는 크롤링/적재 중심으로 설명 가능해야 한다.
- 로컬 개발자도 문서만으로 새 구조를 실행하고 이해할 수 있어야 한다.

## 검증 전략
- 기본 개발 사이클은 Red → Green → Refactor로 유지한다.
- 각 phase 종료 시 문서, 데이터 경로, 사용자 핵심 흐름을 함께 검증한다.
- 프론트 전환 시 최소 `npm run build`, `npm run lint`를 기준 검증으로 둔다.
- 백엔드/크롤러 변경 시 최소 `./gradlew test` 또는 영향 모듈 테스트를 실행한다.
- 병행 운영 단계에서는 동일 기능에 대해 기존 경로와 새 경로 결과를 비교 검증한다.

## 의존성 및 결정 필요 항목
- Next.js 실제 전환 여부와 시점
- 추천 로직 실행 위치
- 크롤러의 Supabase 직접 적재 여부
- Supabase Storage 경로 정책과 공개 범위

## 리스크
- 추천 로직 이전 시 단순 데이터 소스 교체만으로는 동작 품질이 유지되지 않을 수 있다.
- 목록/검색 이전 전에 스키마와 인덱스 설계가 부족하면 성능 이슈가 발생할 수 있다.
- 프론트 프레임워크 전환과 데이터 계층 전환을 동시에 진행하면 범위가 과도하게 커질 수 있다.
