# Implementation Plan

## 상태 규칙
- `[ ]` 미착수
- `[-]` 진행 중
- `[x]` 완료
- 작업 진행 시 체크 상태를 즉시 갱신하고, 완료 기준 충족 시 해당 phase와 세부 작업을 함께 업데이트한다.

## 방향
- 이번 이슈는 기능 추가가 아니라 UI 회귀 복구 작업이다.
- Next.js App Router, 내부 API route, Supabase 조회 구조는 유지하고 프론트 표현 계층만 조정한다.
- Git 히스토리의 `frontend/src/App.tsx`를 기존 경험의 기준점으로 삼되, 추천 기능 관련 흐름은 제외한다.
- 현재 정상 동작 중인 검색, 목록, 무한 스크롤을 깨뜨리지 않는 범위에서 단계적으로 복구한다.

## [-] Phase 1. 기준 UI 분석 및 복구 범위 고정
### 목표
- 기존 카탈로그 UI의 복구 기준을 명확히 하고, 현재 구조에 이식할 범위를 확정한다.

### 작업
- [ ] Git 히스토리에서 삭제된 `frontend/src/App.tsx`와 관련 스타일 구조를 확인한다.
- [ ] 기존 홈 화면의 히어로, 검색 바, 섹션 흐름, 카드 배치, 정보 우선순위를 정리한다.
- [ ] 추천 기능과 결합된 UI 요소를 식별하고 제외 대상을 확정한다.
- [ ] 현재 `CatalogPageClient.tsx`, `LiquorGrid.tsx`, `LiquorCard.tsx`, `globals.css`와 비교해 회귀 항목을 목록화한다.

### 산출물
- 기존 UI 기준점 요약
- 복구 대상 / 제외 대상 목록
- 현재 구조 대비 회귀 체크리스트

### 완료 기준
- 어떤 요소를 얼마나 복구해야 하는지 문서만 봐도 판단 가능해야 한다.
- 추천 관련 영역을 제외한 복구 범위가 확정되어야 한다.

## [ ] Phase 2. 카탈로그 화면 구조 복구
### 목표
- 현재 Next.js 홈 화면에 기존 카탈로그 레이아웃과 화면 흐름을 이식한다.

### 작업
- [ ] `CatalogPageClient.tsx`에 기존 홈 카탈로그 섹션 구조를 반영한다.
- [ ] 검색 UI와 결과 리스트 진입 흐름을 기존 경험에 맞게 조정한다.
- [ ] 현재 데이터 상태 처리 방식과 무한 스크롤 동작을 유지하면서 화면 배치를 재정렬한다.
- [ ] 추천 기능 관련 상태, 버튼, 분기 로직이 다시 유입되지 않도록 확인한다.

### 산출물
- 복구된 카탈로그 메인 UI
- 기존 경험 기준 흐름 정리

### 완료 기준
- 홈 화면의 정보 구조와 레이아웃이 기존 카탈로그 경험에 가깝게 복구되어야 한다.
- 현재 목록/검색 동작은 그대로 유지되어야 한다.

## [ ] Phase 3. 카드/그리드/스타일 회귀 복구
### 목표
- 카드 인상, 간격, 타이포, 전역 스타일을 기존 화면 수준으로 되돌린다.

### 작업
- [ ] `LiquorGrid.tsx`의 배치, 간격, 반응형 레이아웃을 복구한다.
- [ ] `LiquorCard.tsx`의 hover, 정보 노출 방식, 카드 위계를 복구한다.
- [ ] `app/globals.css`에 기존 톤, spacing, typography, 섹션 스타일을 반영한다.
- [ ] 현재 Next.js 구조와 충돌 없이 스타일이 적용되는지 확인한다.

### 산출물
- 복구된 카드/그리드 UI
- 전역 스타일 회귀 복구

### 완료 기준
- 카드와 그리드가 현재 단순 버전이 아니라 기존 카탈로그 인상에 가깝게 보여야 한다.
- 데스크톱과 모바일 모두에서 레이아웃이 크게 깨지지 않아야 한다.

## [ ] Phase 4. 검증 및 문서 마감
### 목표
- UI 복구 결과를 검증하고 문서를 최신 상태로 정리한다.

### 작업
- [ ] `frontend`에서 `npm run lint`를 실행한다.
- [ ] `frontend`에서 `npm run build`를 실행한다.
- [ ] `frontend`에서 `npm run test:e2e`를 실행한다.
- [ ] 가능하면 로컬 실행으로 최종 화면을 확인한다.
- [ ] 검증 결과와 남은 리스크를 `progress.md`에 반영한다.

### 산출물
- 검증 결과 기록
- 최신 진행 문서

### 완료 기준
- 최소 검증 명령이 모두 통과해야 한다.
- 다음 작업자가 현재 상태와 남은 리스크를 문서만으로 이해할 수 있어야 한다.

## 검증 전략
- UI 복구 중에도 검색, 목록 조회, 무한 스크롤 동작을 계속 확인한다.
- 스타일 변경 이후에는 lint/build/e2e를 순차적으로 수행한다.
- 가능하면 개발 서버에서 실제 화면을 확인해 문서 기준 복구 여부를 점검한다.

## 의존성 및 결정 필요 항목
- 없음

## 리스크
- 기존 `App.tsx`를 기준 없이 추정 복구하면 "비슷하지만 다른 화면"이 될 위험이 있다.
- UI 복구 중 현재 정상 동작 중인 검색과 무한 스크롤이 깨질 수 있다.
- 추천 기능 관련 잔여 UI를 무심코 다시 들여오면 현재 제품 방향과 충돌할 수 있다.
