# Lean PRD

## 문제
`001.nextjs-supabase-migration`에서 프론트엔드는 Next.js App Router와 Supabase 기반 조회 구조로 전환되었다. 기능 전환 자체는 성공했지만, 전환 과정에서 기존 Vite 기반 카탈로그 화면의 레이아웃, 시각적 톤, 카드 인상, 화면 흐름이 유지되지 못했다.

현재 홈 화면은 `CatalogPageClient -> LiquorGrid -> LiquorCard` 구조로 재구성되어 목록 조회, 검색, 무한 스크롤은 동작하지만, 사용자가 체감하는 기존 카탈로그 경험은 회귀한 상태다. 이 문제를 해결하지 않으면 기술 마이그레이션은 완료되더라도 실제 서비스 품질은 저하된 상태로 남는다.

## 목표
- Next.js App Router 구조와 Supabase 조회 흐름은 유지한다.
- 기존 카탈로그 UI의 시각적 인상, 레이아웃, 흐름을 가능한 한 가깝게 복구한다.
- 검색, 목록 조회, 무한 스크롤 등 현재 동작 중인 핵심 기능을 유지한 채 UI 회귀를 해소한다.

## 비목표
- 추천 기능을 다시 도입하지 않는다.
- Spring 조회 API를 복구하지 않는다.
- 이번 이슈를 새로운 디자인 리뉴얼이나 브랜딩 개편으로 확장하지 않는다.
- Supabase 쿼리 구조나 데이터 계층을 대규모로 재설계하지 않는다.

## 사용자 가치
- 기존 서비스 이용자가 익숙한 카탈로그 경험을 다시 사용할 수 있다.
- 기술 전환 이후에도 기능뿐 아니라 화면 품질이 유지된다.
- 이후 프론트 개선 작업이 기존 UX 기준점 위에서 안정적으로 이어질 수 있다.

## 핵심 요구사항
- 기존 삭제된 `frontend/src/App.tsx`를 기준으로 카탈로그 UI 레이아웃과 상호작용 요소를 복원해야 한다.
- 추천 관련 흐름은 제외하고 카탈로그 홈, 검색, 목록 영역만 선별적으로 이식해야 한다.
- 현재 Next.js 엔트리와 내부 API 라우트, Supabase 조회 구조는 유지해야 한다.
- `CatalogPageClient.tsx`, `LiquorGrid.tsx`, `LiquorCard.tsx`, `app/globals.css`를 중심으로 UI 회귀를 복구해야 한다.
- 최소 `npm run lint`, `npm run build`, `npm run test:e2e` 검증을 통과해야 한다.

## 성공 기준
- 홈 카탈로그 화면이 기존 Vite 앱의 시각적 인상과 정보 구조에 가깝게 복구된다.
- 검색, 목록 조회, 무한 스크롤이 기존처럼 정상 동작한다.
- 추천 기능이 없는 현재 제품 방향을 유지한다.
- Next.js + Supabase 구조를 유지한 상태로 UI 회귀만 해소된다.
- 관련 문서가 현재 작업 범위와 진행 상태를 정확히 반영한다.

## 범위
포함:
- 삭제된 `App.tsx` 기준 UI 구조 확인
- 카탈로그 관련 UI 요소 선별 복구
- 카드/그리드/전역 스타일 회귀 복구
- 프론트 검증 명령 실행 및 결과 반영

제외:
- 추천 기능 복원
- Spring API 복구
- 데이터 모델 및 쿼리 책임 재설계
- 신규 기능 추가

## 제약 및 전제
- 현재 기능적으로는 Next.js 앱이 정상 동작하며, 이번 이슈는 UI 회귀 복구에 집중한다.
- 삭제된 기존 UI는 Git 히스토리에서 복원 기준을 찾아야 한다.
- 현재 데이터 흐름은 이미 안정화되어 있으므로, UI 복구 과정에서 데이터 fetch 계층 변경은 피해야 한다.
- 기존 UI를 그대로 복제하기보다 현재 구조에 맞게 필요한 영역만 이식해야 한다.
